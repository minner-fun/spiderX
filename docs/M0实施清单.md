# M0 实施清单 — SpiderX 地基

> 配套《SpiderX设计纲要-v1.md》。M0 = 把骨架立起来、三端跑通互通，并把硬约束写进首版数据模型。完成后才进 M1~M4 业务主线。

---

## 0. 里程碑总览

| # | 里程碑 | 产出 | 验收 |
|---|---|---|---|
| **M0** | 骨架 + 基础设施 | monorepo + docker-compose(PG/Redis) + FastAPI/Celery/Vue 三端跑通互通 + 首版数据模型/迁移 | 三端启动无错、健康检查通过、能建一条 Spider 并入库 |
| M1 | 数据模型 + 核心 CRUD | Project/Spider/Version/Run 模型 + 爬虫列表/详情 API + 前端列表页 | 能建爬虫、查列表、看详情 |
| M2 | 规则编辑器 + 试运行 | `POST /rules/preview`：URL+规则→async通用引擎抓首页→返回字段JSON + 四层信号 | 点选规则出真实抽取预览 |
| M3 | 调度 + 执行 + 落库 | Beat(抖动)→enqueue→async worker 执行→幂等入库；Run 状态机 + signals 落库 | 到点自动跑、数据进库、Run 有四层信号 |
| M4 | 实时 + 分诊 | worker→Redis pub-sub→WS→深色分诊看板 + 运行详情实时日志 | 分诊看板看到真实健康三态 |

**M0 只做地基，不做业务逻辑。** 下面是 M0 的具体清单。

## 2. docker-compose（一键起）

服务：
- `postgres:16`（卷持久化，初始库 `spiderx`）
- `redis:7`
- `backend`（uvicorn，依赖 pg+redis，跑 alembic upgrade 后启动）
- `worker`（celery worker + beat，连同一 redis/pg）
- `frontend`（vite dev，代理 /api → backend）

所有连接走 `.env`（`DATABASE_URL` / `REDIS_URL` / `SECRET_KEY` …）。**杜绝旧项目 `local=True` 开关与硬编码密码。**

验收：`docker compose up` 后
- `GET http://localhost:8000/health` → `{db:ok, redis:ok}`
- 前端 `http://localhost:5173` 打开，导航可点
- worker 日志显示已连 broker、Beat 已注册

## 3. 首版数据模型（M0 建表，含硬约束）

M0 先建主线必需 + 把硬约束钉死的表（其余实体 M1+ 再加）：

| 表 | M0 关键字段 | 硬约束 |
|---|---|---|
| `projects` | id, name, env, domain, created_at | 多租户根 |
| `users` | id, name, email, role | 作者归属用 |
| `spiders` | id, project_id, name, owner_id, status, **health_status** default 'unknown', priority, contribution_pct, is_core, current_version_id, created_at | 执行态/健康态**分离** |
| `spider_versions` | id, spider_id, version, rules(JSONB), incremental(JSONB), hooks(JSONB), is_live, author_id, change_msg, created_at | **唯一版本载体**；配置入 JSONB，不落文件 |
| `runs` | id, spider_id, version_id, exec_status, data_outcome, trigger, started_at, finished_at, stats(JSONB), **signals(JSONB)** | **signals 字段必存**（缺层=null=unknown），作唯一真相源 |
| `schedules` | id, spider_id, cron, queue, priority, enabled, jitter_sec, next_run_at, last_run_at | 下发抖动 |
| `dedup_registry` | scope(inque/crawled), spider_id, key, expire_at | **入队+已采两套**，带 TTL |

> 设计纲要 §4 的其余实体（HealthSnapshot/SnoozeState/AuthorMetric/ScheduleReconcile/ProxyPool/CredentialPool/AlertRule/LogEntry）在 M1~M4 按需引入，但 `runs.signals`、`spider_versions.rules`、`spiders.health_status` 这三个硬约束 **M0 就位**，避免后期返工。

Alembic 初始迁移建上述表 + 一条 seed（1 个 project + 1 个 user）。

## 4. M0 要写的最小代码

**backend**
- `core/config.py`：从 env 读取，Pydantic Settings。
- `core/db.py`：async engine + session 依赖。
- `core/redis.py`：async redis 客户端。
- `main.py`：`/health`（查 db+redis）+ WS `/ws/ping`（echo，验证实时链路）。
- `models/`：§3 七张表的 SQLAlchemy 定义 + Alembic 首迁移。
- `api/spiders.py`：`POST /api/projects/:pid/spiders` + `GET` 列表（最小 CRUD，验证 PG 链路）。

**worker**
- `celery_app.py`：Celery(app) 连 Redis broker + Beat。
- `tasks.py`：`ping_task` —— 写一条 `runs`（带占位 signals）+ 发 Redis pub-sub 消息。**验证「调度→worker→入库→实时推送」全链路。**
- `sink/pg_sink.py`：`upsert(records)` 骨架（M0 仅打通连接）。

**frontend**
- 侧边导航 + 顶栏（线框 app shell）。
- `views/Triage.vue`：占位三态卡 + 「系统健康」面板（调 `/health`）。
- `api/`：axios 封装，跑通一次 `GET /api/.../spiders`。

**验收主链路（M0 终验）**：前端建一个 Spider → 后端入 PG → 触发 `ping_task` → worker 写一条 Run + pub-sub → 前端 WS 收到消息。**这条打通，地基就成了。**

## 5. M0 明确不做（避免范围蔓延）
- 不做真实抓取引擎（M2）。
- 不做规则编辑器 UI（M2）。
- 不做分诊算法/分层信号计算（M3/M4，M0 只把 `signals` 字段留好）。
- 不做代理池/并发管控/反爬（后续）。
- 不做 RBAC/告警/作者考评屏（M1+/二期）。

---

## 6. 启动顺序建议
1. 建 monorepo 骨架 + `.env.example` + docker-compose。
2. 起 PG/Redis，跑通 backend `/health`。
3. Alembic 首迁移建 §3 七表 + seed。
4. 打通 worker `ping_task` 全链路（含 pub-sub→WS）。
5. 前端 app shell + Triage 占位 + 一次真实 API 调用。
6. M0 终验主链路 → 进 M1。
