# 项目背景与设计沉淀

> 本文把 SpiderX 立项前对 **6 个真实旧项目**的复盘、以及由此逼出的所有设计决策，沉淀成可追溯的来龙去脉。读它能理解「为什么这么设计」。配套：`docs/SpiderX设计纲要-v1.md`（最终设计）、`CLAUDE.md`（速览）。

---

## 一、来历

用户做爬虫多年，手上一摞新老项目。SpiderX 是**全新平台**：旧项目全部废弃、不做兼容，但把它们的痛与经验吸收进新设计。设计经历：看 v1 草图 → 复盘 6 个旧项目 → 三轮设计反馈（design_handoff_v1/v2/v3 + hifi）→ 架构逐项讨论 → 收敛成纲要 + M0 清单 → M0 落地。

业务输入（用户拍板）：
- 业务价值开发者手填，但 ~20 个站贡献 80% 数据量 → 核心站重点盯梢。
- 主数据库 PostgreSQL；业务数据可插拔 sink，MVP 默认落 PG。
- 配置驱动 + 代码驱动都要，配置优先。
- 部署先单机（测试机 sh-2c2g，2c2g），按量扩。
- 代理：3500/月套餐，每分钟拉 30 个 IP 进 Redis、1 分钟过期、随机取；另有隧道代理每请求换 IP。

## 二、6 个旧项目复盘（反面/正面需求来源）

### 项目光谱总览
| 档位 | 代表项目 | 给 SpiderX 的 |
|---|---|---|
| 配置驱动（简单） | 招投标 Scrapy ×1485、jobs Next.js API | 1模板+N配置、统一sink、标准schema |
| 配置驱动 v0（已验证） | modules_http RPA | 可行性 + 要补的治理/队列/健康欠账 |
| 代码驱动（复杂） | jianyu、boss | 显式DAG、双去重、cookie旁路、JS签名钩子 |
| 反面教训 | Muto、IC | 丢数据、复制文件当版本、并发瓶颈 |
| 成熟 worker 框架 | Sardinia（别团队） | 4 角色分离 + URL 路由的范本 |

### 1. Muto（celery+rabbitmq，已废弃）— 9 个维护痛点
`D:\qifumin\Muto`。痛点逐条对应 SpiderX 的反向需求：
1. 配置/密钥全硬编码散落（broker带密码写死、MySQL密码写死、redis多实例写死），切环境靠 `local=True` 全局开关。
2. 队列手写、三处同步（50+ 个 Queue() 手写，新增阶段要改队列声明+task+.conf）。
3. Worker 拓扑写死 systemd（doc/ 近 30 个 .conf，队列→节点→并发映射写死，扩容靠改文件+restart）。
4. 发任务靠手跑巨型脚本（send_task.py 2227 行硬编码数据，SSH 上去 python 执行，无 UI 无调度）。
5. 大文件+注释当版本（jianyu.py 3242 行，大量注释掉的历史版本）。
6. 无可观测性（日志写磁盘，看进度靠 SSH tail）。
7. 无 Run 记录、重试自我放大（异常时 send_task 给自己，可雪崩）。
8. 状态散在 Redis（去重 sadd、页码存 redis、多实例手连）。
9. 关注点混杂（kafka/对象存储上传/验证码/代理池全内联进爬虫）。
→ **SpiderX 三支点对症**：配置驱动(消灭代码增殖)、分层信号(自动诊断)、审计+归属(消灭人盯人)。

### 2. ic_website_spider（IC 电子元器件，celery，已废弃）— 两个实锤
`D:\qifumin\ic_website_spider`。
- **丢数据根因**：app/ic_app.py `worker_prefetch_multiplier=4` 但**没有** `task_acks_late`/`task_reject_on_worker_lost`。Celery 默认 early-ack（任务一到就 ack，不等执行成功），prefetch=4 即每进程预取4条立即全 ack；worker 崩溃→这4条已被删但没执行→无日志。正是用户说的「确认进队列了然后没了、日志查不到」。
- **复制文件当版本**：plantform/ 下同一个 tti 爬虫 11 个副本（_备份/_拆分/_细化0225…各~1万行），整项目15万行多为重复备份。没人知道线上跑哪版。
→ SpiderX：**late-ack + DLQ + Run 记录作唯一真相源 + 对账器**；**SpiderVersion 版本化**。

### 3. 招投标 Scrapy（~1485 爬虫，已废弃）— 配置驱动论断的实证
`D:\qifumin\ppc\lins-文件件\bid_project_three(1400) + bid_project_four(85)`。
- ~1485 爬虫共享**完全相同**的：26 字段 schema(BidProjectThreeItem)、同一 pipeline 链、同一 settings、同一代理中间件。每爬虫唯一不同：列表URL+日期参数、列表行selector、字段selector、翻页、详情selector + 小转换(万元→元、地区→省市区)。= 字面的「1模板+N配置」。
- **他们的 pipeline 链 = 我们的统一 Sink 设计且已实战**：归一化(uuid=sign(url))→PushToSelenium(硬站推Redis给渲染服务)→对象存储(html/pdf/image)→MysqlPool(REPLACE/INSERT IGNORE 幂等去重)→Kafka(带version/trace_sn信封)→Redis(doc去重)。
- 可直接复用：26 字段招投标 schema、tools.py 标准转换(sign/judge_area/judge_notice_type/get_date)+4400行省市区行政区划表。
- 仍存的坑：依然 1485 手写文件、headers copy-paste、增量是「固定回采近3天+去重」非水位线、无健康自省、local 开关。
→ 证明配置驱动可把 N 份代码收敛成 1 模板 + N 配置；招投标域 schema/转换可作起点。

### 4. modules_http RPA（电商，自造，2026 在产）— 最重要的先验
`D:\qifumin\lora\trunk\modules_http`(+ ImportData)。**用户已造过的配置驱动平台 v0**。
- DB 驱动 3 段：createTask(建任务)→startTask(执行)→ImportData(解析入库)。核心 generate_http.py 的 MyHttp = 完全由 MySQL 配置表驱动的通用 HTTP 引擎：请求头/URL/body 是配置行；字段解析配置行(tiers JSON路径如 data.$.list + respFunc)；签名/加密=DB里存的代码片段 exec_py/exec_js；任务状态机 WAIT/RUNNING/ERROR/FINISHED+retry+锁。
- **已端到端验证我们设计的一大半**：配置驱动请求、代码钩子、浏览器兜底、cookie 会话管理、列表→详情=父子任务派生、翻页+停止条件+值模板、每段状态机+卡住重试。
- **但精确暴露 SpiderX 要补的欠账**：①配置=不透明DB行+列里塞可执行代码(无版本/难测/不安全) ②调度器=MySQL轮询LIMIT 1+巨型UNION+手写锁 ③对账=updatedOn<60min就重试 粗糙 ④自创JSON提取小语言重发明JSONPath ⑤密码硬编码+注释切环境 ⑥无集群健康/分诊。
→ SpiderX = 接着这套往下走，补：受治理版本化配置、真队列、标准提取、健康分诊、可靠性。其配置表结构是设计 schema 的真实参考。

### 5. jianyu / boss（复杂代码驱动）— 平台原语
- **jianyu**（剑鱼历史标书，celery）：复杂在队列编排——多入口(智略词/搜索/需求页)汇流→detail→clean→[zoo,kfk]；撞登录墙→派 captor_cookies 旁路任务刷 cookie+重投；**双去重闸门** in_quee(入队去重防爆炸)+crawled_id(已采去重)，还靠手动脚本每天清旧条目。
- **boss**（直聘，最硬反爬）：execjs 跑网站自己的 boss.js 算签名，从 cookie 抠 __zp_sseed__ 种子→算签名；**sign 与 cookie 耦合**。
- jobs-scraping：抓 Next.js `_next/data/{buildId}/...json` 数据接口绕过 HTML（配置驱动最好吃）；那个「自定义下载器」其实是空模板。
→ SpiderX 代码驱动档需要的平台原语：**显式 DAG/流水线**(不散落 send_task)、**双去重闸门**(带TTL)、**凭据池+异步刷新旁路**、**JS-exec 钩子(沙箱+读cookie)**、多目标 sink。

### 6. Sardinia（别团队 2022 TS 框架，~71 硬站）— worker 架构范本
`D:\qifumin\ppc\lins-文件件\sardiniacrawler`。TS + @koios/sardinia + MongoDB，目标全是国内最硬反爬站(工商/爱企查/裁判文书/药监局/BOSS/电商)。
- 声明式插件：crawler_plugin.ts 声明 crawlerJob(任务源) + pageDownloader(怎么抓) + pageProcessors:[{urlReg, processorName}](**URL正则→解析器**)。worker 层拆 4 角色：JobSource/Downloader/Processor/Reaper(收割入库,独立角色)。工商用位掩码控制 28 个子数据类型选择性抓。
- **验证**：4 角色分离 = 我们执行器抽象；URL正则→规则路由比散落 callback 干净，SpiderX 采纳；Reaper 独立 = 抓取与入库解耦。
- **但不是平台**：是「写爬虫的框架」非「管 2000 爬虫的平台」，无调度/健康/版本/多租户；每站仍手写 TS 插件，无配置即数据。栈是 Node/TS/Mongo，**借概念不搬代码**。

## 三、设计决策与对应的旧项目证据（速查）

| SpiderX 决策 | 来自哪个项目的痛/经验 |
|---|---|
| 配置即数据(1模板+N配置) | 招投标1485份代码、Sardinia手写插件 |
| 四层信号健康分诊 | 用户复盘「改版挂了 vs 真没数据」分不清 |
| 核心站盯梢(20站80%数据) | 用户运维经验(帕累托) |
| late-ack + DLQ + Run对账 | IC early-ack 丢数据实锤 |
| SpiderVersion 版本化 | IC 复制文件11副本、Muto注释当版本 |
| 显式DAG + 双去重闸门 | jianyu 队列编排 + in_quee/crawled |
| 凭据池 + 异步刷新旁路 | jianyu captor_cookies 旁路 |
| JS-exec 钩子(读cookie) | boss execjs + __zp_sseed__ 签名 |
| URL正则→Processor路由 | Sardinia pageProcessors |
| 4角色执行器 + 抓取入库解耦 | Sardinia / modules_http ImportData |
| 受治理配置(非DB存裸代码) | modules_http exec_py/exec_js 反面 |
| 真队列(非MySQL轮询) | modules_http LIMIT 1+UNION 调度反面 |
| 代理双driver + IP感知并发 | 用户代理现状(30滚动IP/隧道) |
| 集中密钥(无local开关) | Muto/IC/modules_http 反复硬编码 |
| 作者考评=去重持续出数据站点数 | 实习生拿旧站冒充新站糊弄 |
